# Slide Archetypes

Use this file when defining the deck’s repeatable system.

## 1. Cover

Purpose:

- frame the meeting
- state the value proposition or the desired decision

Pattern:

- eyebrow
- strong title
- short subtitle
- one supporting proof cluster or visual panel

Avoid:

- long disclaimers
- visible process language
- presenter notes on the canvas

## 2. Why Now

Purpose:

- explain the business pressure or platform inflection point

Pattern:

- decisive title
- short context line
- `2-4` pressure cards
- optional audience-facing implication bar for leave-behind decks; otherwise keep the implication in presenter notes

Avoid:

- too many sub-issues on one slide
- body copy that spills below the cards
- visible takeaway bands that are really speaker notes

## 3. Recommendation

Purpose:

- say what OCI approach is recommended

Pattern:

- recommendation headline
- optional summary or guardrail bar only when the audience needs it visible; presenter-led default is notes
- `3` supporting pillars or tradeoff cards

Avoid:

- hedged titles
- a generic service list

## 4. Proof / Value

Purpose:

- support the recommendation with business or platform evidence

Pattern:

- statement headline
- `2-4` proof blocks
- optional full-width outcome bars with a fixed title column and explicit `1-2` line bodies
- optional guardrail or implication strip only when it is audience-facing rather than presenter-only

Avoid:

- repeating the same card layout too many times in a row
- long single-line bar copy that depends on implicit text wrapping
- mixing unrelated proof types on one slide

## 5. Architecture

Purpose:

- show the target-state platform or workload view

Pattern:

- one architecture only
- short explanatory subtitle
- `2-4` callouts or takeaway cards

Avoid:

- multiple diagrams on one slide
- text labels resting on traffic lanes
- crowding the diagram with deck copy that belongs elsewhere

## 6. Roadmap

Purpose:

- show the path from starting point to validated progress

Pattern:

- clear phase headline
- `3` phases or workstreams
- optional visible takeaway or dependency note for leave-behind decks; presenter-led default is notes

Avoid:

- too many milestones
- phase boxes with inconsistent size or padding
- footer notes that read like presenter coaching

## 7. Next Step

Purpose:

- make the ask explicit

Pattern:

- clear next-step headline
- optional visible output bar when the deck must stand alone without a presenter
- `3` action blocks
- one closing CTA

Avoid:

- vague asks
- ending the deck without an owner, decision, or workshop outcome
- visible workshop-output bands that belong in presenter notes
